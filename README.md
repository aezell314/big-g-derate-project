# Big G Express: Predicting Full Derates
 
A machine learning project focused on predicting severe truck derates using J1939 fault code and onboard diagnostic (OBD) telemetry data from heavy-duty commercial vehicles.
 
---
 
# Overview
 
This project analyzes fault events and onboard diagnostic measurements from a fleet of heavy-duty trucks to predict upcoming **full derates** — critical engine protection events that severely limit vehicle operation and often require towing and emergency repairs.
 
The primary objective was not simply maximizing model accuracy, but maximizing **business value** by minimizing operational costs associated with missed derates and unnecessary maintenance events.
 
---
 
# Business Problem
 
A **full derate** occurs when a truck’s onboard computer intentionally restricts engine power to protect the vehicle from damage. In this dataset, full derates are identified by:
 
- **SPN = 5246**
- `active = True`
 
## Estimated Model Costs
 
| Event Type | Estimated Cost |
|---|---|
| Predicted full derate | $4,000 gain due to avoided tow and maintenance |
| False positive prediction | $500 loss due to unnecessary maintenance |
 
---
 
# Project Goals
 
- Predict a full derate **at least 2 hours before it occurs**
- Compare predictive performance across:
  - 6-hour prediction window
  - 12-hour prediction window
  - 24-hour prediction window
- Maximize overall cost savings
- Identify the most predictive fault and telemetry signals

# Dataset Description
 
## 1. J1939 Fault Data 
 
Contains standardized fault code events generated by vehicle ECUs.
 
### Key Fields
 
| Column | Description |
|---|---|
| `EventTimeStamp` | Time of fault occurrence |
| `spn` | Suspect Parameter Number |
| `fmi` | Failure Mode Identifier |
| `active` | Whether the fault became active |
| `EquipmentID` | Unique truck identifier |
| `Latitude` / `Longitude` | Vehicle location |
| `RecordID` | Unique fault event ID |
 
---
 
## 2. Diagnostic Telemetry 
 
Contains onboard vehicle measurements linked to fault events through `FaultId`.
 
### Example Signals
 
- Engine RPM
- Engine oil pressure
- Fuel rate
- Turbo boost pressure
- Speed
- Coolant temperature

# Filtering Service Station Events
 
Faults occurring near maintenance locations were excluded because trucks being actively serviced may generate misleading fault activity.
 
## Service Locations
 
- `(36.0666667, -86.4347222)`
- `(35.5883333, -86.4438888)`
- `(36.1950, -83.174722)`
 
## Approach
 
- Used a vectorized Haversine distance calculation
- Removed all records within **1 mile** of a service station
 
## Result
 
Approximately **11%** of records were removed.

# Handling Missing Data
 
The dataset contained extensive missing telemetry values.
 
## Strategies Used
 
- Forward-fill for cumulative counters
- `IterativeImputer` for numeric features
- `SimpleImputer(strategy="most_frequent")` for categorical features
 
## Scaling
 
We used `MaxAbsScaler` to scale numerical values because many features were sparse and we could not assume they were normally distributed. 
 
---
 
# Model Training
 
# Validation Strategy
 
We used a 5-fold chronological cross-validation approach where training folds always occurred before validation folds chronologically. This prevented data leakage and simulated real-world deployment conditions.
 
---
 
# Class Imbalance Handling
 
Two balancing approaches were evaluated.
 
## 1. Upweighting Minority Class
 
Used XGBoost’s:
 
```python
class_weight="balanced"
```
 
---
 
## 2. SMOTE Oversampling
 
Applied synthetic minority oversampling to training data.

# Key Challenges
 
## Severe Class Imbalance
 
Derates were extremely rare relative to normal fault activity.
 
## Missing Data
 
Many telemetry columns were missing for 50%+ of records.
 
## Weak Long-Term Signals
 
Strong indicators existed close to derates (<2 hours), but predictive power weakened significantly outside that window.
 
## Operational Drift
 
Several external factors likely impacted model quality:
 
- Fleet modernization
- Operational changes at Big G (such as improved maintenance practices)
- Big G’s own internal early-warning system was introduced mid-dataset
 
---
 
# Key Takeaways
 
- Predicting full derates more than 2 hours in advance is extremely difficult with the available data.
- Operational business metrics can paint a very different picture than traditional ML metrics.
- More sophisticated feature engineering may improve performance.
- Additional telemetry consistency and maintenance history would likely improve the predictive power of this model.

# How to use this repo

Big G Express did not give permission for the full dataset to be shared publicly; as a result, the Python code will not run using the public repository materials alone. 

However, the repository was structured such that if the data was present, anyone interested in replicating this project would be able to run main.py and generate the same model and metrics that are presented in this readme.

The requirements.txt file contains the Python packages used in this project. Run the following on your local terminal to install all required packages:

```python
pip install -r requirements.txt
```

Once all required packages are installed, run the main Python script:
```python
python3 main.py
```
